Overview

From chaos to clarity

Product teams spend hours on intake calls only to produce vague, incomplete feature requests. PRISM automates the entire process with AI agents that know what questions to ask.

The Problem

Feature requests arrive as Slack messages, emails, and meeting notes — unstructured, incomplete, and missing critical details. PMs spend hours in follow-up calls, and specs still ship with gaps that derail engineering sprints.

The Solution

PRISM deploys a team of specialized AI agents that interview stakeholders, evaluate completeness in real-time, and produce structured specifications that meet a rigorous Definition of Ready — all in a single conversation.

Architecture

Collaborative agent loop

Three core agents work in a continuous feedback loop across multiple LLM providers. The Critic silently evaluates every exchange, feeding signals back to the Investigator until the spec is complete.

SSE Streaming

Responses stream in real-time via Server-Sent Events, so users see the investigator "typing" as it formulates questions.

Silent Critic

The Critic agent runs in parallel and never speaks to users. It evaluates conversations and feeds explore/flag/contradiction signals to the Investigator.

Adaptive Depth

Feature complexity is classified (simple/medium/complex) and scoring thresholds adjust dynamically so simple features resolve faster.

PII Detection

User messages are scanned for personally identifiable information which is flagged and can be redacted before ticket generation.

Duplicate Detection

New requests are compared against recent interviews via semantic similarity to warn users before duplicating existing work.

Tech Neutrality

The Critic enforces technology-neutral language. If a user dictates implementation ("use React"), it's flagged and rephrased as a requirement.

🎯 Submission Threshold

18 / 27

Total score must reach at least 18 out of 27 points. All required criteria must individually score ≥ 2.

🛡️ Tech Neutrality Gate

Must be 3

Tech Neutrality must score exactly 3/3. The system actively detects and rephrases technology-dictating language.

🔄 Escalation Rule

After 3 failed rounds on the same criterion, the system escalates with direct, targeted questions to resolve the gap.

⚡ Critic Signals

The Critic emits real-time signals: explore (need more info), flag (issue detected), contradiction, PII detected, off-topic, and scope creep.

Quality Gates

Only tickets with DoR scores ≥ configurable threshold (default 18/27) are embedded. Pilot-org tickets and manually excluded tickets are filtered out to prevent noise.

Embedding Text

Each ticket is serialized into a structured text block including feature description, business rationale, requirements, scope boundaries, and product area. SHA-256 hash prevents redundant re-embedding.

Organization Scoping

Similarity search is scoped to the current organization, ensuring data isolation in the multi-tenant architecture. Each org only sees its own past ticket patterns.

Experimentation

Prompt A/B testing

Every agent prompt can be A/B tested. The system uses deterministic assignment to ensure consistent experiences and measures real outcomes.

Step 1

Create Experiment

Define variant A (control) and variant B (challenger) prompts for any agent. Set traffic split percentage (default 50/50).

Step 2

Deterministic Assignment

Each interview is assigned a variant using SHA-256 hashing of the test ID + interview ID. Same inputs always yield the same variant.

hash = SHA256(testId + interviewId)
bucket = hash[0:8] mod 100
variant = bucket < split ? "A" : "B"
Step 3

Measure Outcomes

Track critic scores, revision counts, interview rounds, and completion rates per variant to identify which prompts perform better.
